#### Q3：ClickHouse 为什么快？你项目里怎么用的？

```
为什么快：
1. 列式存储：只读需要的列，减少 I/O
2. 稀疏索引：自动按 ORDER BY 建索引，快速定位
3. 向量化执行：SIMD 指令批量处理数据
4. 数据压缩：列式存储压缩比高（10:1）
5. ReplacingMergeTree：自动去重，不用手动 DISTINCT

我的使用：
- 表设计：ReplacingMergeTree，ORDER BY (city, date, element)
- 查询：WHERE city='北京' AND date='2024-01-01' → 稀疏索引命中
- 去重：后台合并时自动按 ORDER BY key 去重
- 数据量：3000+城市，千万级数据点/天

为什么不用 MySQL：
- MySQL 行存，查询需要全表扫描
- ClickHouse 列存，只读需要的列
- 同样查询：MySQL 3秒 → ClickHouse 50ms
```

**面试官可能追问**：
- ReplacingMergeTree 的去重原理？（后台 merge 时保留版本最大的行）
- 实时写入怎么保证查询到最新数据？（FINAL 关键字 或 版本号过滤）

#### Q5：异步 LLM 调用怎么做的？为什么 54 倍提速？

```
问题：
- 同步调用：每个请求等 LLM 返回（1-3秒）
- 2600+ 场景 × 3秒 = 2小时+

解决方案：asyncio + aiohttp + Semaphore(50)

import asyncio
import aiohttp

async def call_llm(prompt, semaphore):
    async with semaphore:  # 控制并发数，避免打爆 API
        async with aiohttp.ClientSession() as session:
            async with session.post(LLM_URL, json={"prompt": prompt}) as resp:
                return await resp.json()

async def batch_process(prompts):
    semaphore = asyncio.Semaphore(50)  # 最大 50 并发
    tasks = [call_llm(p, semaphore) for p in prompts]
    results = await asyncio.gather(*tasks)
    return results

# 2600 个任务 / 50 并发 = 52 批
# 每批 3 秒 → 总计 52 × 3 = 156 秒 ≈ 2.5 分钟
# 对比同步：2600 × 3 = 7800 秒 ≈ 2 小时
# 提速：7800 / 156 = 50 倍（实际 54 倍，因为有些请求更快）
```

**面试官可能追问**：
- Semaphore 的作用？（控制并发数，防止 API 限流）
- 如果某个请求超时怎么办？（asyncio.wait_for + timeout）
- 结果顺序怎么保证？（asyncio.gather 返回顺序与输入一致）

### 项目四：实时数据缓存系统（C++）

#### Q8：为什么用 C++ 不用 Redis？

```
场景特点：
- 单机缓存 200 万数据点（1~4GB）
- 查询延迟要求极低（<1ms）
- 数据是气象时序数据，格式固定

Redis 的问题：
- 网络开销：每次查询要走 TCP（0.5-1ms）
- 序列化开销：JSON 序列化/反序列化
- 内存效率：Redis 对象头开销大

C++ 内存缓存的优势：
- 零网络开销：直接内存访问（纳秒级）
- 自定义编码：数据点→uint64 key，O(1) 哈希查找
- 内存效率：紧凑存储，无对象头开销
- shared_mutex：读写分离，多线程并发读

适用场景：
- 单机、数据量可控、延迟要求极高
- 不需要持久化（可从快照恢复）
```

**面试官可能追问**：
- shared_mutex 怎么实现读写分离？（读锁共享，写锁独占）
- 快照怎么做的？（定时序列化 → zstd 压缩 → 写磁盘）
- 数据一致性怎么保证？（写时复制 / 版本号）

#### Q9：二进制协议设计？

```
请求协议（9字节）：
┌─────────┬─────────┬─────────┬─────────┐
│ cmd(1B) │ key(4B) │ ver(4B) │         │
│ 命令类型 │ 数据键   │ 版本号   │         │
└─────────┴─────────┴─────────┴─────────┘

响应协议（6字节）：
┌─────────┬─────────┬─────────┐
│ code(1B)│ len(2B) │ data(3B)│
│ 状态码   │ 数据长度  │ 数据    │
└─────────┴─────────┴─────────┘

为什么用二进制不用 JSON：
- JSON：{"cmd":1,"key":12345} → 20+ 字节
- 二进制：01 00 00 30 39 → 9 字节
- 解析速度：JSON 需要解析，二进制直接 memcpy
- 网络带宽：节省 50%+

ZMQ REQ/REP：
- 同步请求响应模式
- 比 HTTP 轻量（无 Header）
- 适合内部服务通信
```

**面试官可能追问**：
- 为什么不用 gRPC？（ZMQ 更轻量，内部服务不需要 HTTP/2）

### 项目六：数据质量监控与智能异常检测

#### Q11：异常检测算法怎么选的？

```
三层检测体系：

1. 统计检测（基础）
   - z-score：假设正态分布，|x - μ| > 3σ 为异常
   - IQR：不假设分布，Q1 - 1.5*IQR 或 Q3 + 1.5*IQR 为异常
   - MAD：中位数绝对偏差，对异常值鲁棒

   自动选择：
   if 数据正态性检验通过:
       用 z-score
   else:
       用 IQR 或 MAD

2. 趋势检测（进阶）
   - 时序模型（ARIMA/Prophet）预测
   - 残差 = 实际值 - 预测值
   - 残差突变 → 异常

3. 极端边界分析（高级）
   - KMeans 聚类：正常数据聚一类，异常数据聚一类
   - 边界点 = 故障阈值
   - 对比人工设定：更客观、更准确

成果：异常检出率提升40%，漏报率<5%
```

**面试官可能追问**：
- z-score 和 IQR 什么时候用？（正态用 z-score，非正态用 IQR）
- KMeans 怎么确定 K 值？（肘部法则 / 业务经验）
- 实时检测怎么做？（增量计算统计量，窗口滑动）

#### Q13：遇到过最难的技术问题是什么？怎么解决的？

```
问题：Flink 流处理上线后，部分数据丢失

排查过程：
1. 对比批处理和流处理结果，发现流处理少了 3% 数据
2. 检查 Kafka 消费日志，发现 Consumer Group rebalance 频繁
3. 分析原因：Flink TaskManager 频繁重启，触发 rebalance
4. 根因：K8s 资源限制太紧，OOM 导致 Pod 被杀

解决方案：
1. 调整 K8s 资源限制（内存从 2G → 4G）
2. Flink checkpoint 间隔从 1min → 30s（减少数据丢失窗口）
3. Kafka offset 提交改为 checkpoint 成功后提交
4. 增加数据对账：批处理每天对账一次，发现丢失自动补数据

成果：数据丢失率从 3% → 0.01%（可接受范围）
```
