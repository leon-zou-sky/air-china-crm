## 三、面试题

### 3.1 会员积分系统

#### Q1：为什么积分扣减用乐观锁而不是悲观锁？

**标准答案**：

```
场景特点：
- 积分查询:扣减 = 100:1（读多写少）
- 同一会员同时扣减概率低（写冲突少）

悲观锁（SELECT FOR UPDATE）：
- 查询时锁行，其他事务等待
- 100次查询都要加锁解锁，性能差
- 适合写冲突频繁场景

乐观锁（version字段CAS）：
- 查询不加锁，扣减时才检查version
- 100次查询零开销，冲突时才重试
- 适合读多写少场景

实现：
UPDATE t_miles_account 
SET balance = balance - #{miles}, version = version + 1
WHERE member_id = #{memberId} AND version = #{version}

影响行数=0 → 版本冲突 → 重试（最多3次）
```

**追问**：
- 乐观锁的ABA问题怎么解决？（版本号单调递增）
- 重试3次还是失败怎么办？（抛异常，返回用户重试）
- 分布式环境下乐观锁有什么问题？（数据库压力大，可用Redis分布式锁）

#### Q2：Cache Aside 模式的工作原理？

**标准答案**：

```
读取流程：
1. 先查 Redis 缓存
2. 命中 → 直接返回（<1ms）
3. 未命中 → 查 MySQL → 写入 Redis → 返回（50ms）

更新流程：
1. 先更新 MySQL
2. 再删除 Redis 缓存（不是更新缓存）
3. 下次读取时重新从 MySQL 加载

为什么删除而不是更新缓存：
- 避免并发更新时缓存和数据库不一致
- 删除缓存成本更低
- 下次读取时惰性加载

容错设计：
- Redis 不可用时降级查库（try-catch）
- 保证核心业务不受缓存影响
```

**追问**：
- 缓存和数据库数据不一致怎么办？（延迟双删）
- 缓存穿透怎么解决？（布隆过滤器/空值缓存）
- 缓存雪崩怎么解决？（随机TTL/限流降级）

#### Q3：规则引擎的设计与实现？

**标准答案**：

```
规则维度：
- 会员等级：GENERAL/SILVER/GOLD/PLATINUM
- 舱位：F（头等）/C（公务）/Y（经济）
- 变更类型：CHANGE/REFUND

匹配策略（优先级从高到低）：
1. 精确匹配：会员等级 + 舱位 + 变更类型
2. 模糊匹配：会员等级 + 变更类型（舱位为空=通用）
3. 默认规则：兜底规则

实现：
public class TicketRuleEngine {
    public Rule match(String tier, String cabin, String changeType) {
        // 按优先级降序排列，第一个匹配的规则生效
        return rules.stream()
            .filter(r -> r.match(tier, cabin, changeType))
            .findFirst()
            .orElse(defaultRule);
    }
}

规则存储在数据库，支持动态配置，新增规则无需改代码
```

### 3.2 工单服务系统

#### Q4：状态机模式在工单系统中的应用？

**标准答案**：

```
状态定义：
CREATED → ASSIGNED → IN_PROGRESS → COMPLETED → CLOSED

合法流转：
FROM            TO                  触发场景
──────────────────────────────────────────────────
CREATED         ASSIGNED            客服指派
CREATED         CLOSED              客服取消
ASSIGNED        IN_PROGRESS         地服开始处理
ASSIGNED        CLOSED              地服拒绝
IN_PROGRESS     COMPLETED           服务完成
IN_PROGRESS     CLOSED              异常关闭
COMPLETED       CLOSED              归档

实现：
public class WorkOrderFlowEngine {
    // 定义合法流转
    private static Map<String, Set<String>> TRANSITIONS = Map.of(
        "CREATED", Set.of("ASSIGNED", "CLOSED"),
        "ASSIGNED", Set.of("IN_PROGRESS", "CLOSED"),
        "IN_PROGRESS", Set.of("COMPLETED", "CLOSED"),
        "COMPLETED", Set.of("CLOSED")
    );
    
    public void transit(String fromStatus, String toStatus) {
        if (!TRANSITIONS.get(fromStatus).contains(toStatus)) {
            throw new BizException("非法状态流转：" + fromStatus + " → " + toStatus);
        }
    }
}
```

**追问**：
- 如果要新增一个状态怎么办？（修改TRANSITIONS映射，不影响其他代码）
- 状态机和工作流引擎（如Activiti）的区别？（状态机简单轻量，适合固定流程；工作流引擎适合复杂审批流）
- 如何保证状态流转的事务性？（状态更新和流转记录在同一个事务）

#### Q5：SLA 超时监控怎么实现？

**标准答案**：

```
定时任务：@Scheduled(fixedRate = 5分钟)

扫描逻辑：
1. 查询所有活跃工单（CREATED/ASSIGNED/IN_PROGRESS）
2. 计算剩余时间 = SLA截止时间 - 当前时间
3. 更新SLA状态：
   - 剩余 > 30分钟 → NORMAL（正常）
   - 剩余 ≤ 30分钟 → AT_RISK（即将超时，WARN日志）
   - 已过截止时间 → BREACHED（已超时，ERROR日志）

告警方式（可插拔设计）：
- 当前：日志打印（WARN/ERROR）
- 后续：接入MQ → 钉钉/邮件告警

查询接口：
- GET /api/workorder/sla/at-risk    即将超时
- GET /api/workorder/sla/breached   已超时
- GET /api/workorder/sla/abnormal   所有异常
```

**追问**：
- 定时任务多实例部署会重复执行吗？（需要分布式锁，如Redis SETNX）
- SLA时间怎么计算？（创建时间 + 模板配置的小时数）
- 告警频率怎么控制？（避免重复告警，可以加告警状态标记）

#### Q6：工单模板配置化怎么设计？

**标准答案**：

```
模板表结构：
CREATE TABLE t_work_order_template (
    template_id    INT PRIMARY KEY,
    service_type   VARCHAR(32),   -- 服务类型
    template_name  VARCHAR(64),   -- 模板名称
    flow_nodes     JSON,          -- 流转节点
    sla_hours      INT,           -- SLA时限（小时）
    default_priority TINYINT      -- 默认优先级
);

示例数据：
service_type=LOUNGE, sla_hours=2, flow_nodes=["CREATED","ASSIGNED","IN_PROGRESS","COMPLETED","CLOSED"]
service_type=WHEELCHAIR, sla_hours=3, flow_nodes=["CREATED","ASSIGNED","IN_PROGRESS","COMPLETED","CLOSED"]

创建工单时：
1. 根据 service_type 查询模板
2. 从模板获取 SLA 时限，计算截止时间
3. 从模板获取流转节点，校验状态流转合法性

新增服务类型：只需插入一条模板记录，无需改代码
```

### 3.3 通用问题

#### Q7：RabbitMQ 如何保证消息可靠性？

**标准答案**：

```
四个环节：
1. Producer → Exchange：mandatory=true + ConfirmCallback
2. Exchange → Queue：durable=true（持久化）
3. Queue 存储：durable=true（持久化）
4. Queue → Consumer：手动ACK（acknowledge-mode=manual）

异常处理：
- 消费失败：重试3次 → 进入死信队列（DLX）
- 死信队列：人工处理或告警

本项目应用：
- 积分变动 → sms.queue（短信通知）
- 工单派发 → pek/sha/can.queue（按机场路由）
- SLA告警 → alert.queue（钉钉/邮件）
```

#### Q8：项目中用了哪些设计模式？

**标准答案**：

```
1. 乐观锁模式（Optimistic Locking）
   - 应用：积分扣减
   - 实现：version字段CAS
   - 场景：读多写少、写冲突少

2. 状态机模式（State Machine）
   - 应用：工单流转
   - 实现：枚举定义合法状态转换
   - 场景：固定流程的状态管理

3. 规则引擎模式（Rule Engine）
   - 应用：退改签费用计算
   - 实现：优先级匹配（精确→模糊→默认）
   - 场景：多维度规则匹配

4. 门面模式（Facade）
   - 应用：客户360视图
   - 实现：聚合多个Service的数据
   - 场景：对外提供统一查询接口

5. Cache Aside 模式
   - 应用：Redis缓存
   - 实现：读缓存→未命中查DB→写缓存
   - 场景：高频读、低频写

6. 接口隔离模式
   - 应用：Service接口+Impl分离
   - 实现：面向接口编程
   - 场景：为后续微服务拆分预留
```
